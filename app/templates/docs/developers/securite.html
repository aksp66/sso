Sécurité & bonnes pratiques

Recommandations pour intégrer Nexus de façon sécurisée dans votre application.

Toujours utiliser PKCE

Même pour les clients confidentiels, PKCE ajoute une couche de protection supplémentaire contre l'interception de code. Nexus l'exige pour tout client public — il est fortement recommandé même pour les clients serveur.

Valider systématiquement le state

Ne traitez jamais une redirection vers votre redirect_uri sans vérifier que le state reçu correspond exactement à celui généré avant la requête /authorize.

Ne jamais exposer le client_secret côté client

Si votre application tourne dans un navigateur ou sur un appareil mobile, enregistrez-la comme client public (sans secret) et utilisez PKCE. Le client_secret ne doit exister que côté serveur.

Stocker les tokens en sécurité

  • Access token et ID token : mémoire ou stockage de courte durée, jamais en localStorage pour les SPA exposées au XSS
  • Refresh token : cookie HttpOnly + Secure si possible, ou stockage serveur

Valider la signature et l'expiration des tokens

Récupérez les clés publiques via /jwks.json, identifiez la bonne clé via le claim kid de l'en-tête JWT, et vérifiez la signature RS256 ainsi que exp avant de faire confiance au token.

Révoquer les tokens à la déconnexion

Appelez POST /revoke avec l'access token et/ou le refresh token lorsque l'utilisateur se déconnecte de votre application, afin qu'ils soient immédiatement invalidés côté Nexus.

Tous les endpoints Nexus doivent être appelés en HTTPS en production. Le serveur ajoute Strict-Transport-Security et refuse tout contenu mixte via sa Content-Security-Policy.
